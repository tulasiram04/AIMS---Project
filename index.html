Project Overview

What is AIMS?

AIMS is an enterprise-grade, AI-first platform that autonomously reconciles discovered infrastructure assets against CMDB records, identifies governance anomalies, assesses compliance risk, and delivers executive-level intelligence — all orchestrated through five specialized AI agents.

📌
The Core Problem
Enterprise IT environments drift continuously. Assets are provisioned without CMDB registration, configuration changes go untracked, and naming conventions erode over time. Reconciliation is manual, periodic, and error-prone — creating dangerous blind spots that surface only during audits.

CMDB Drift Rogue Assets Audit Failures Compliance Gaps
✅
The AIMS Solution
AIMS deploys a coordinated swarm of five autonomous AI agents that ingest live infrastructure inventory, compare it against CMDB expected state, classify every discrepancy with Gemini-powered root cause analysis, generate remediation plans, and synthesize everything into governance scores and executive PDF reports — continuously and automatically.

Autonomous Continuous AI-Driven Executive-Ready
🏢
Who It's Built For
Large enterprises running ServiceNow CMDB alongside hybrid cloud, on-premise data centers, and multi-tenant SaaS environments. IT operations teams, compliance officers, and C-suite executives who need asset truth at scale.
⚡
How It Works
Upload your current infrastructure inventory (CSV/JSON) alongside ServiceNow CMDB exports. AIMS orchestrates all five agents sequentially, producing a full reconciliation report, AI analysis, governance scores, and downloadable executive PDF within minutes.
🎯
The Governance Promise
Every reconciliation cycle produces a quantified Governance Score (0–100), Inventory Health Score, Compliance Readiness Rating, and Trust Score — giving leadership an unambiguous measure of their IT estate's integrity.
Business Challenge

Why Enterprises Fail at Asset Governance

Five systemic failures create cascading risk across IT, compliance, and finance functions. AIMS was designed to address each one directly.

🔍
HIGH RISK
Missing Assets
Assets that exist in the CMDB as expected — servers, VMs, network devices — but have gone offline, been decommissioned, or simply disappeared without formal change management. During audits, these create unexplainable gaps.
→ Impact
SOC 2 audit findings, SLA breaches, security blind spots
→ AIMS Response
Flags every CMDB-expected asset with no discovered counterpart. Auto-creates incident tickets via ServiceNow integration.
👻
MEDIUM RISK
Untracked (Rogue) Assets
Assets discovered on the live network that have no corresponding CMDB record. These are often shadow IT deployments, unauthorized cloud VMs, or legacy systems that were never formally registered.
→ Impact
Unpatched attack surface, license violations, capacity miscalculations
→ AIMS Response
Discovery agent surface all unregistered assets with AI-generated CMDB entry recommendations.
⚙️
MEDIUM RISK
Configuration Drift
Assets that match by identity but whose actual configuration (OS version, RAM, CPU, software) no longer matches the CMDB record. The most common and hardest-to-detect form of data quality failure.
→ Impact
Change management failures, patch gaps, capacity planning errors
→ AIMS Response
Attribute-by-attribute comparison with confidence scoring and AI-generated update scripts.
🏷️
LOW RISK
Naming Convention Drift
Assets with hostnames, identifiers, or labels that deviate from the enterprise naming standard. While seemingly minor, naming inconsistencies break automation toolchains, monitoring dashboards, and cross-system integrations.
→ AIMS Response
Fuzzy-match detection using RapidFuzz with similarity scoring. Rename recommendations generated automatically.
📋
COMPLIANCE
Compliance & Audit Failures
Regulatory frameworks (SOC 2, ISO 27001, GDPR, HIPAA, PCI-DSS) require documented, accurate asset inventories. Manual reconciliation cycles are too slow and too infrequent to maintain continuous compliance posture.
→ AIMS Response
Continuous compliance scoring with per-framework mapping and audit-ready PDF evidence packages.
🔁
OPERATIONAL
Operational Inefficiency
Traditional reconciliation requires a team of analysts, days of manual comparison work, and specialist knowledge of CMDB schemas. With estates of 50,000+ assets, this is economically unsustainable and cognitively error-prone.
→ AIMS Response
Full reconciliation cycle completed autonomously in minutes, not days. Zero manual comparison required.

Multi-Agent Architecture

Five Specialized AI Agents

Each agent owns a discrete domain of the reconciliation pipeline. Agents communicate through a shared context object managed by the LangGraph-style orchestrator, passing enriched state downstream at each stage.

01 INGESTION LAYER

Inventory Ingestion Agent

The entry point for all infrastructure data. This agent handles the complexity of diverse data formats from multiple sources, normalizing everything into a canonical AssetRecord schema before passing to the reconciliation pipeline. It is the data quality guardian of the entire system.

  • Parse CSV files from network scanners (Nmap, Qualys, Tenable) and CMDB exports
  • Parse JSON payloads from cloud APIs (AWS Config, Azure Resource Graph, GCP Asset Inventory)
  • Normalize asset attributes to canonical schema — hostname, IP, MAC, OS, owner, environment
  • Validate data completeness, flag malformed records, and compute initial data quality score
  • Deduplicate assets by composite key (hostname + IP + MAC) with configurable precedence rules
Input / Output Contract
INPUT: raw_csv_path: str raw_json_path: str cmdb_export_path: str OUTPUT: normalized_assets: List[AssetRecord] data_quality_score: float rejected_records: List[str] ingestion_metadata: dict
CSV Parser JSON Parser Pydantic Validation RapidFuzz Dedup
02 RECONCILIATION LAYER

Reconciliation Agent

The analytical core of AIMS. This agent performs exhaustive asset matching between the discovered inventory and the expected CMDB state, classifying every asset into one of six reconciliation outcomes using a multi-pass comparison algorithm.

  • Exact match: hostname, IP, MAC, and all attributes align — asset confirmed healthy
  • Missing asset detection: expected CMDB asset has no discovered counterpart
  • Untracked asset detection: discovered asset has no CMDB record — potential rogue
  • Naming drift detection: fuzzy hostname matching (RapidFuzz ≥85% similarity threshold)
  • Configuration drift: attribute-level diff — OS, RAM, CPU, software versions, tags
  • Duplicate detection: multiple assets resolving to same identity with conflicting data
Matching Algorithm
PASS 1: Exact match by (hostname ∧ IP ∧ MAC) PASS 2: Fuzzy match hostname similarity = RapidFuzz.ratio(h1, h2) threshold = 0.85 PASS 3: IP-only match (hostname changed) PASS 4: MAC-only match (IP reassigned) UNMATCHED → Missing | Untracked
RapidFuzz Multi-Pass Attribute Diff Confidence Score
03 AI ANALYSIS LAYER

AI Analysis Agent

The intelligence layer. This agent consumes the raw reconciliation results and calls Gemini 1.5 Flash to perform contextual root cause analysis, business impact assessment, and governance evaluation that no rule-based system could produce.

  • Root cause analysis: WHY did this discrepancy occur? (unauthorized change, decommission without CMDB update, migration, etc.)
  • Business impact analysis: What is the financial, operational, and compliance risk of this asset discrepancy?
  • Confidence scoring: Assigns 0–100 confidence to each reconciliation finding based on context
  • Governance assessment: Maps each issue to CIS Controls, NIST CSF, SOC 2, ISO 27001 frameworks
  • Predictive drift: Identifies assets trending toward non-compliance based on historical patterns
Gemini Integration
MODEL: gemini-1.5-flash-latest METHOD: Structured prompt engineering + asset diff context injection + enterprise knowledge base OUTPUT: root_cause: str business_impact: str confidence: float [0–1] frameworks_affected: list remediation_priority: int
Gemini 1.5 Flash Root Cause AI Impact Analysis MCP Protocol
04 REMEDIATION LAYER

Recommendation Agent

Converts analysis findings into actionable remediation plans. This agent generates specific, executable corrective actions for each discrepancy type — prioritized by business risk and sequenced for minimal operational disruption.

  • Generate step-by-step remediation runbooks for each discrepancy type
  • Prioritize corrective actions by risk score × blast radius calculation
  • Draft ServiceNow change requests in ITIL-compliant format for CMDB updates
  • Generate compliance gap remediation plans mapped to specific control requirements
  • Estimate remediation effort, cost, and timeline for planning purposes
Priority Calculation
Priority Score = (Risk Severity × 0.4) + (Compliance Impact × 0.3) + (Asset Criticality × 0.2) + (Remediation Ease × 0.1) P1 (≥80): Immediate Action Required P2 (60–79): This Sprint P3 (40–59): This Quarter P4 (<40): Backlog
ITIL Aligned Risk-Ranked ServiceNow API
05 EXECUTIVE LAYER

Executive Summary Agent

The final stage of the pipeline. This agent synthesizes the outputs of all previous agents into board-ready governance intelligence: scored metrics, visualized trends, and professionally formatted PDF executive reports suitable for CISOs, CIOs, and audit committees.

  • Compute composite Governance Score from weighted inventory health, compliance, and trust metrics
  • Generate multi-page executive PDF with branded layouts using ReportLab
  • Produce trend analysis comparing current reconciliation against historical baselines
  • Create audit evidence packages with full data lineage and agent execution logs
  • Broadcast real-time governance metrics to Prometheus for Grafana dashboard alerting

Reconciliation Engine

Six Reconciliation Outcomes

Every asset in a reconciliation cycle is classified into one of six outcome categories. Each category triggers a specific analysis path and remediation workflow.

✓ EXACT MATCH
Exact Match
Asset exists in both discovered inventory and CMDB. All attributes — hostname, IP, MAC, OS, hardware spec — match exactly. No action required. Asset contributes positively to governance score.
CMDB: prod-web-01 | 10.0.1.10 | Ubuntu 22.04 | 16GB LIVE: prod-web-01 | 10.0.1.10 | Ubuntu 22.04 | 16GB STATUS: ✓ HEALTHY
⚠ MISSING
Missing Asset
Asset is registered in CMDB as expected but was not found during live discovery. Could indicate unauthorized decommission, hardware failure, network isolation, or VM termination without change management.
CMDB: prod-db-backup | 10.0.5.22 | Windows Server 2019 LIVE: [NOT FOUND - 0 matching records] STATUS: ⚠ MISSING — P1 Incident Created
👁 UNTRACKED
Untracked Asset
Asset discovered on live network but has no CMDB record. Classic shadow IT or rogue device scenario. Security risk: unpatched, unmanaged, potentially unauthorized. Requires immediate investigation.
CMDB: [NO RECORD FOUND] LIVE: unknown-vm-47 | 10.0.8.99 | CentOS 7 | 8GB STATUS: 👁 UNTRACKED — CMDB Entry Recommended
~ NAMING DRIFT
Naming Mismatch
Asset matched by IP/MAC but hostname deviates from CMDB record. Fuzzy matching identifies the relationship with a similarity score. Breaks monitoring tools and automation that rely on hostname-based discovery.
CMDB: prod-app-server-01 | 10.0.2.15 LIVE: prd-appsvr-01 | 10.0.2.15 MATCH: RapidFuzz similarity = 0.87 → NAMING DRIFT
⚙ CONFIG DRIFT
Configuration Drift
Asset identity matches CMDB but one or more configuration attributes differ. Most common finding type. Could indicate unauthorized change, failed patch, or infrastructure scaling event that bypassed change management.
CMDB: prod-db-01 | RAM: 32GB | OS: RHEL 8.4 LIVE: prod-db-01 | RAM: 64GB | OS: RHEL 8.7 DRIFT: RAM +32GB, OS patch level +3 versions
⊕ DUPLICATE
Duplicate Asset
Multiple CMDB records or discovered instances resolve to the same physical/virtual asset. Indicates data quality failures in the CMDB. Creates inventory count inflation and misleads capacity planning.
CMDB: prod-web-03 | 10.0.3.5 | ID: CI0001234 CMDB: production-web-3 | 10.0.3.5 | ID: CI0001891 LIVE: prod-web-03 | 10.0.3.5 STATUS: ⊕ DUPLICATE CMDB ENTRIES — Merge Required
Governance Engine

Governance Scoring Model

AIMS computes five quantitative governance metrics that together form the composite Governance Score. Each metric is defined by a transparent, auditable formula.

📊 Inventory Health Score
98.2
IHS = (Exact Matches / Total Expected Assets) × 100 − (Missing Assets × 5) − (Config Drifts × 2) + (Successfully Remediated × 1)
Measures the raw accuracy of the current inventory state against the CMDB expected baseline. The primary indicator of CMDB data quality.
📋 Compliance Score
99.1
CS = Σ(framework_weights × framework_pass_rate) where frameworks = [SOC2, ISO27001, CIS, NIST, PCI] and framework_weights sum to 1.0
Weighted composite of compliance status across all enabled regulatory frameworks. Weight reflects business relevance (configurable per organization).
🏆 Governance Score
94.7
GS = (IHS × 0.30) + (CS × 0.30) + (TS × 0.20) + (RS × 0.20) where: IHS = Inventory Health Score CS = Compliance Score TS = Trust Score RS = 100 − Risk Score
The single headline governance metric. A composite of all four sub-scores with configurable weights. Tracked historically to show governance improvement or degradation over time.

🔒 Trust Score
TS = (Data Quality × 0.40) + (Source Confidence × 0.35) + (Historical Accuracy × 0.25)
Measures the reliability of the data sources feeding into AIMS. Low trust scores indicate unreliable discovery tools or stale CMDB data that should be refreshed before acting on reconciliation results.
⚠️ Risk Score
RS = Σ(issue_severity × asset_criticality × exposure_score × age_factor) normalized to [0, 100]
Quantifies the total risk exposure from all open reconciliation issues. Accounts for severity, affected asset criticality, network exposure, and time since detection. Lower is better; ≤20 is the target.

Access Control

Role-Based Access Control

Security Implementation
JWT access tokens (15 min expiry) + refresh tokens (7 day, rotated on use) · bcrypt password hashing (cost=12) · CSRF protection via SameSite cookies · Rate limiting: 5 failed logins → 15-minute lockout · All auth events logged to audit_logs table with IP and user agent · Passwords shown as demo placeholders only — actual values are hashed on seed.
